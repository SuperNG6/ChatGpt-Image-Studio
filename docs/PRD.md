# ChatGpt Image Studio 产品需求文档

版本：v0.1  
日期：2026-05-10  
状态：基于当前代码现状整理，供后续迭代使用

## 1. 产品概述

ChatGpt Image Studio 是一个面向图片创作与账号池管理的单服务工作台。项目通过 Go 后端统一承载图片接口、账号池、配置、同步、任务队列与静态前端，前端提供图片生成/编辑、对话辅助创作、账号管理、配置管理和请求观测能力。

当前交付形态为“一个后端服务 + 一份静态前端 + 本地或远端持久化配置”，既可本地运行，也可通过 Docker 部署到具备持久化存储或 Redis 的环境。

## 2. 背景与问题

图片生成工作流通常同时面临几个问题：

- 图片创作需要反复生成、编辑、参考图迭代和历史回溯，纯接口调用体验不够完整。
- 多账号环境下，需要管理额度、账号状态、账号来源、同步关系和不同链路的可用性。
- Free / Plus / Pro / Team 账号在分辨率、路由能力和上游链路上存在差异，需要更明确的调度策略。
- 本地部署、容器部署和无状态云部署对数据存储有不同要求，需要在文件、SQLite、Redis、浏览器存储之间灵活切换。
- 当请求失败时，用户需要知道是账号、代理、路由、队列、上游还是配置问题，而不是只看到泛化错误。

本项目的核心价值是把“图片创作工作台”和“多来源账号/链路运维”放在同一个可自托管产品里，让用户能以更低成本完成持续图片创作。

## 3. 目标用户

### 3.1 创作用户

需要高频生成、编辑图片，关注提示词沉淀、参考图连续编辑、历史记录、失败重试和结果下载。

### 3.2 账号池维护者

维护多个 ChatGPT 账号、Token 账号或远端号池来源，关注额度刷新、账号状态、同步、禁用、优先级和分组使用策略。

### 3.3 自部署用户

需要在本机、NAS、服务器、Docker 或无状态云平台部署，关注配置持久化、代理、Redis、SQLite、诊断和升级。

### 3.4 接口兼容用户

希望通过 `/v1/images/*`、`/v1/chat/completions`、`/v1/responses` 等兼容接口接入已有工具链，同时复用本项目的账号池与图片缓存能力。

## 4. 产品定位

ChatGpt Image Studio 不是单纯的图片 Demo，也不是只提供 API 代理。它定位为：

- 图片创作侧：可持续迭代的图片工作台。
- 账号运营侧：面向图片额度和多来源同步的账号池管理后台。
- 部署运维侧：可自托管、可观测、可诊断、可配置迁移的单服务应用。
- 接口兼容侧：对外提供 OpenAI 风格图片与对话接口，降低外部工具接入成本。

## 5. 当前产品能力

### 5.1 图片工作台

当前已支持：

- 文本生图。
- 上传参考图生成。
- 图片编辑。
- 选区涂抹式局部重绘。
- 多轮图片会话历史。
- 新建会话、继续编辑、失败重试、提示词复制、结果图下载。
- 按比例选择分辨率，区分 Free 实际档和 Paid 高分辨率档。
- Low / Medium / High 质量选择。
- 图片任务队列、运行状态、取消任务、实时任务流更新。
- 移动端历史页与工作台页分离。
- 图片历史可使用浏览器存储或服务端存储。

### 5.2 对话模式

当前已支持：

- 本地保存对话历史。
- 发送文本消息。
- 上传图片作为对话上下文。
- 上传可读文本类文档并截取内容作为上下文。
- 使用账号池中的可用账号发起对话。
- 将多轮对话整理为图片生成提示词，并跳转到图片任务。

### 5.3 账号管理

当前已支持：

- 导入本地认证文件。
- Studio 模式直接导入 `access_token`。
- 区分认证文件账号和 Token 账号。
- 展示账号类型、状态、额度、恢复时间、同步状态等信息。
- 单账号刷新额度。
- 批量刷新额度，并展示刷新进度。
- 删除、禁用、更新账号信息。
- 图片账号路由策略：排序、分组、启用分组、额度保留比例。
- CPA / NewAPI / Sub2API 多来源同步与推送。

### 5.4 配置管理

当前已支持：

- 页面化编辑 `data/config.toml` 相关配置。
- 图片模式切换：Studio / CPA。
- Free / Paid 图片路由配置：legacy / responses。
- Free / Paid 上游模型配置。
- CPA route strategy 配置：`images_api` / `codex_responses` / `auto`。
- 代理配置与测试。
- NewAPI / Sub2API 集成配置与探测。
- 账号池存储、配置存储、图片会话和图片数据存储配置。
- 文件、SQLite、Redis 等存储后端。
- 运行时配置热更新，并在关键存储配置变更时迁移数据。

### 5.5 请求与诊断

当前已支持：

- 图片请求记录，最多保留最近 200 条。
- 记录 endpoint、operation、imageMode、direction、route、CPA 子路由、排队耗时、账号信息、模型、size、quality、promptLength、错误码等。
- 启动检查：服务、代理、官方链路、账号、CPA 等。
- 运行时状态：队列、账号、近期失败、图片任务状态。
- 诊断信息导出。

### 5.6 OpenAI 兼容接口

当前已支持：

- `POST /v1/images/generations`
- `POST /v1/images/edits`
- `POST /v1/chat/completions`
- `POST /v1/responses`
- `GET /v1/models`
- `GET /v1/files/image/{filename}`

## 6. 产品目标

### 6.1 短期目标

- 稳定图片工作台的生成、编辑、重试、取消和历史体验。
- 补齐对话模式到图片工作台的闭环，让“先聊清楚再生图”成为顺滑主流程。
- 提升账号池可解释性，让用户明确知道为什么某个任务排队、失败或选用了某类账号。
- 固化部署和诊断流程，减少新用户首次启动失败和配置错误。

### 6.2 中期目标

- 提供更强的创作资产管理能力，包括收藏、标签、搜索、批量导出、提示词模板。
- 提供更完整的任务中心，包括批量任务、任务优先级、任务恢复、失败原因聚类。
- 提供更可靠的账号调度策略，包括健康评分、冷却、成本/额度策略、手动指定账号组。
- 提供面向多人或轻团队的权限、共享历史和审计能力。

### 6.3 长期目标

- 成为一个可自托管的 AI 图片创作与账号资源调度平台。
- 支持更多模型来源和图片能力，如多模型路由、批量风格测试、工作流模板、提示词评估。
- 提供稳定的插件化或集成接口，让外部工具可以复用图片任务、历史和账号池能力。

## 7. 核心使用流程

### 7.1 首次部署流程

1. 用户启动服务。
2. 系统生成默认配置。
3. 用户进入登录页并输入管理认证。
4. 用户进入启动检查页查看代理、账号、链路状态。
5. 用户导入账号或配置 CPA / NewAPI / Sub2API。
6. 用户进入图片工作台开始生成。

### 7.2 图片生成流程

1. 用户输入提示词。
2. 选择生成模式、比例、分辨率档、质量和数量。
3. 可选上传参考图。
4. 系统创建图片任务。
5. 任务进入队列，按全局并发、账号可用性、Paid 需求、来源账号约束进行调度。
6. 前端通过任务流展示排队、运行、成功、失败或取消。
7. 用户下载结果、继续编辑、局部重绘或重试失败图片。

### 7.3 对话转图片流程

1. 用户在对话模式描述目标，可上传图片或文档。
2. 系统使用可用账号返回创作建议或提示词方向。
3. 用户点击生成图片。
4. 系统汇总多轮对话，生成结构化图片提示词。
5. 图片任务保存到图片会话历史，并进入工作台。

### 7.4 账号维护流程

1. 用户导入认证文件或 Token。
2. 系统识别账号类型、状态和图片额度。
3. 用户刷新额度或批量刷新。
4. 用户配置账号路由策略、禁用异常账号或调整优先级。
5. 如配置远端来源，可执行同步或推送。

## 9. 非功能需求

### 9.1 性能

- 图片任务创建接口应在正常情况下 1 秒内返回任务状态。
- 图片任务执行受上游耗时影响，但前端必须持续展示排队或运行状态。
- 请求日志保留最近 200 条，避免内存无限增长。
- 前端历史列表应避免因大量 base64 图片导致卡顿，服务端图片存储模式应优先使用 URL 引用。

### 9.2 可用性

- 当没有可用账号、没有 Paid 账号、来源账号不可用、队列满或代理不可达时，应提供明确错误。
- 配置页保存前应尽可能校验关键字段。
- 启动检查应给出可执行修复提示。
- 移动端应保留核心图片工作流。

### 9.3 安全

- UI 管理接口必须经过管理认证。
- 图片兼容接口必须经过图片 API Key 认证。
- 诊断导出需要脱敏敏感配置。
- access_token、management key、API key、Redis 密码等不应在前端或日志中明文泄露。

### 9.4 可维护性

- 后端继续保持 API handler、账号存储、配置、同步、任务管理、上游 client 分层。
- 前端保持页面、组件、store、API client 分层。
- 新增功能应优先补充 Go 单元测试和前端核心工具函数测试。

## 10. 数据与存储

当前涉及的数据：

- 配置数据：默认配置 + 覆盖配置，可落文件或 Redis。
- 账号数据：认证文件、Token、运行状态、同步状态、图片路由策略，可落文件、SQLite 或 Redis。
- 图片任务数据：当前为后端内存任务队列，完成后短期保留。
- 图片会话历史：浏览器 localforage 或服务端 imagehistory store。
- 图片文件：服务端 `image_dir` 或浏览器 data URL。
- 对话历史：当前浏览器 localforage。
- 请求记录：当前后端内存环形列表。
- Flow 画布数据：画布、节点、连线、节点关联的图片任务和图片历史引用。

后续建议：

- 将任务历史和请求日志支持可选持久化，便于排障。
- 对话历史增加服务端存储模式，与图片历史保持一致。
- Flow 画布优先存储图片引用和节点元数据，避免重复保存大图。
- 建立统一的数据导入导出格式，覆盖图片历史、对话历史、提示词模板和配置快照。

## 11. 技术架构摘要

### 11.1 后端

- 语言：Go。
- HTTP 路由：标准库 `http.ServeMux`。
- 核心模块：
  - `backend/api`：路由、handler、图片任务、请求日志、诊断、配置、同步。
  - `backend/handler`：ChatGPT 官方链路 client、Responses client。
  - `backend/internal/accounts`：账号模型、存储、租约、路由策略、同步状态。
  - `backend/internal/config`：配置加载、默认值、环境变量。
  - `backend/internal/imagehistory`：图片会话服务端存储。
  - `backend/internal/*api`：CPA / NewAPI / Sub2API 集成 client。

### 11.2 前端

- 技术栈：Vite + React 19 + TypeScript + Tailwind CSS + localforage + lucide-react。
- 核心页面：
  - `/image/history`、`/image/workspace`：图片工作台。
  - `/chat`：对话模式。
  - `/flow`：无限画布创作模式，规划中。
  - `/accounts`：账号管理。
  - `/settings`：配置管理。
  - `/requests`：请求记录。
  - `/startup-check`：启动检查。
- 核心 store：
  - `image-conversations.ts`：图片历史。
  - `chat-conversations.ts`：对话历史。
  - `flow-canvases.ts`：Flow 画布、节点和连线，规划中。
  - `auth.ts`：UI 登录态。
  - `image-account-policy.ts`：图片账号策略。

### 11.3 Flow 技术选型建议

优先评估 tldraw 作为无限画布底座。tldraw 是面向 React 的无限画布 SDK，适合承载自定义图片节点、文本节点、自由布局和移动端/桌面端交互。

备选方案：

- React Flow：更适合严格节点流程编排，如果 Flow 模式后续更偏“工作流 DAG”，可以采用或借鉴其节点/边数据模型。
- Excalidraw：适合手绘白板和轻量标注，但若需要深度定制图片任务节点、状态同步和生成操作，扩展成本可能更高。

建议 MVP 使用 tldraw 风格的自由画布体验，同时在业务数据层保留 `nodes + edges` 的稳定结构，避免未来从自由画布迁移到流程编排时重写业务模型。

## 12. 关键指标

### 12.1 产品指标

- 图片任务成功率。
- 图片任务平均等待时长。
- 图片任务平均完成时长。
- 失败任务重试成功率。
- 日活跃图片会话数。
- 对话转图片任务占比。
- 历史图片复用编辑比例。

### 12.2 运维指标

- 可用图片账号数。
- 可用 Paid 账号数。
- 账号刷新成功率。
- 最近 10 分钟失败数。

## 14. 风险与待确认问题

- 官方链路和模型能力会变化，Free / Paid 路由策略需要持续验证。
- 部分 Paid 默认链路在 README 中已注明需要真实账号端到端验证，后续应建立验证清单。
- 浏览器存储大图会占用较多本地空间，长期使用需要更明确的容量提示和清理机制。
- Redis / SQLite / 文件三套存储并存，会增加迁移和故障恢复复杂度。
- Token 账号、认证文件账号、远端同步账号之间的边界需要在 UI 中继续解释清楚。
- 多用户能力尚未设计，当前管理认证更适合个人或可信内网部署。
- Flow 模式会引入更复杂的前端状态和大图渲染压力，需要优先控制节点数量、图片懒加载和画布数据体积。

## 17. 术语表

- Studio 模式：直接使用本地账号池访问官方 ChatGPT 链路。
- CPA 模式：通过 CPA 配置的远端服务进行图片请求。
- Legacy 路由：项目原有 `/backend-api/conversation` 图片流程。
- Responses 路由：官方 `/backend-api/codex/responses` 流程。
- Paid 账号：Plus / Pro / Team 类型账号。
- 图片任务：由工作台创建的异步图片生成或编辑任务。
- 来源账号：某张原始图片生成时使用的账号，连续编辑时可能需要复用。
- Flow 模式：基于无限画布的图片创作模式，以节点和连线组织提示词、图片、编辑和生成分支。
